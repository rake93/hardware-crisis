Training gets the headlines, but inference determines whether AI is economically viable. This paper by Xiaoyu Ma and David Patterson at Google DeepMind lays out why current hardware is a poor fit for LLM inference — and proposes four research directions to fix it.

The Problem

Why LLM Inference Is Getting Harder

Six trends in modern LLMs are compounding the resource demands of inference, pushing memory, bandwidth, and communication to their limits.

🧩

Mixture of Experts

Models like DeepSeek-V3 use 256 experts. Sparsity helps training costs but massively expands memory and communication for inference.

🧠

Reasoning Models

Chain-of-thought generates long internal "thinking" sequences before answering, increasing latency and straining memory with extra tokens.

🎬

Multimodal

Text → image, audio, and video generation. Larger data types demand far more than text alone.

📜

Long Context

Longer context windows improve quality but drive up both compute and memory demands significantly.

🔍

RAG

Retrieval-Augmented Generation adds external knowledge as extra context — useful but resource-hungry.

🌊

Diffusion

Generates all tokens at once then iteratively denoises. Unlike others, this mainly increases compute rather than memory.

Inference Basics

The Two Phases of LLM Inference

Transformer inference has two fundamentally different phases — and they bottleneck on different hardware resources.

⚡
Compute Bound

Prefill

Processes all input tokens simultaneously, similar to training. Inherently parallel and typically limited by compute FLOPS.

🔗
Memory Bound

Decode

Generates tokens one at a time (autoregressive). Each step reads the entire KV Cache from memory, making it bandwidth-limited.

Decode Challenge #1

The Memory Wall

Compute has scaled dramatically faster than memory bandwidth. From 2012 to 2022, NVIDIA GPU FLOPS grew 80× while bandwidth grew only 17×. This gap continues to widen, making memory-bound Decode increasingly inefficient.

HBM is getting more expensive, not cheaper. The normalized cost per GB and cost per GB/s of HBM grew 1.35× from 2023–2025, while standard DDR4 costs dropped to roughly half. Manufacturing complexity of HBM stacks is driving this divergence.

DRAM density scaling is also decelerating — a 4× capacity jump that used to take 3–6 years now takes over 10. And SRAM-only approaches (Cerebras, Groq) have proven insufficient as LLM model sizes rapidly outgrew on-chip capacity.

Decode Challenge #2

End-to-End Latency

Unlike training (which runs for weeks), inference must respond in seconds or less. Two latency metrics matter:

Time-to-Completion

Longer outputs = longer latency. Each Decode step also has high memory access latency since it's memory-bound. Long inputs slow it further via KV Cache access.

Time-to-First-Token

Long inputs, RAG, and reasoning models all increase the work before the first visible output token, stretching this critical metric.

Interconnect latency now trumps bandwidth for inference. Before LLMs, inference ran on a single chip. Now it requires multi-chip systems with frequent, small messages — where latency matters far more than throughput. MoE and long-context models make this worse by increasing system size.

The Solutions

Four Research Directions to Rethink Inference Hardware

The paper proposes four synergistic research opportunities. All improve performance per TCO, per watt, and per CO₂e. They can be combined in a single architecture.

01
💾

High Bandwidth Flash (HBF)

HBF stacks flash memory dies like HBM to combine HBM-level bandwidth with flash-level capacity. One HBF stack delivers 512 GB at ~1,638 GB/s read bandwidth — 10× the capacity of an HBM4 stack at comparable bandwidth.

  • 10× weight memory — Host giant MoE models that are unaffordable today
  • 10× context memory — Store slow-changing data like web corpora, code databases, and paper archives
  • Smaller systems — Fewer chips needed, reducing network overhead and power
  • Sustainable scaling — Flash capacity doubles every ~3 years while DRAM growth decelerates

Two key limitations: limited write endurance (HBF is best for read-heavy data like frozen weights) and high read latency at page granularity (microseconds vs. nanoseconds for DRAM). HBF complements HBM; it can't fully replace it.

⚙️

Processing-Near-Memory (PNM)

Rather than placing logic inside memory dies (PIM), PNM keeps memory and compute on separate but nearby dies. This avoids PIM's painful tradeoffs while still boosting bandwidth 2–5× over standard memory.

PIM vs. PNM: Why PNM Wins for Datacenter LLMs
Dimension PIM PNM Winner
Data movement power Very low (on-chip) Low (nearby) PIM
Bandwidth/Watt 5–10× standard 2–5× standard PIM
Logic performance Slower (DRAM process) Full logic process PNM
Memory density Worse (shared with logic) Not affected PNM
Software sharding 32–64 MB banks 16–32 GB shards PNM
Commodity pricing No (custom fab) Yes (standard memory) PNM
Power/thermal budget Tight on DRAM die Less constrained PNM

The key advantage: PNM shards can be 1,000× larger than PIM bank sizes, making LLM partitioning far easier with lower communication overhead. PIM may still be viable for mobile, where energy constraints dominate and models are much smaller.

03
🏗️

3D Memory-Logic Stacking

3D stacking uses vertical through-silicon vias (TSVs) to create a wide, dense memory interface — delivering high bandwidth at low power by shortening data paths. Two flavors exist:

  • Compute-on-HBM-base-die — Reuses HBM designs by inserting compute logic into the HBM base layer. Same bandwidth as HBM but 2–3× lower power.
  • Custom 3D solutions — Go beyond HBM with wider/denser interfaces and advanced packaging for bandwidth exceeding standard HBM.

Challenges include thermal management (less surface area for cooling) and the need for an industry-standard memory interface. However, Decode's low arithmetic intensity means the compute die can run at low clock/voltage, easing thermal concerns.

04
🌐

Low-Latency Interconnect

Current datacenter interconnects optimize for bandwidth (designed for training). LLM inference needs a latency-first redesign — frequent, small messages in a large network.

  • High-connectivity topologies — Tree, dragonfly, and high-dimensional tori reduce hop counts and latency at the possible expense of bandwidth
  • Processing-in-network — In-network acceleration for broadcast, all-reduce, and MoE dispatch/collect operations
  • AI chip optimization — Store small packets directly in on-chip SRAM; place compute engines near the network interface
  • Reliability co-design — Local standby spares reduce failure recovery latency; "approximate communication" uses prior results when messages time out

Beyond Datacenter

What About Mobile?

Mobile devices face different constraints — no HBM, extreme energy limits, single-user batch sizes, and smaller models. The paper notes that PIM (less practical for datacenter) may actually work well on mobile, where models have fewer weights, shorter context, and simplified sharding needs. HBF and 3D stacking also have mobile-optimized variants worth exploring.

The Takeaway

Current AI Hardware Is a Mismatch

The prevailing approach — full-reticle dies with maximum FLOPS, many HBM stacks, and bandwidth-optimized interconnect — was designed for training, not for the memory-bound, latency-sensitive reality of LLM Decode inference.

The paper argues for a fundamental shift: instead of optimizing compute, focus on memory capacity, memory bandwidth, and network latency. New performance metrics that account for total cost of ownership, power, and carbon footprint should guide system design.

The authors also call for a roofline-based performance simulator that the academic community can use to explore these directions — similar to how branch prediction and cache research thrived when realistic simulators became available.

As inference chip sales are projected to grow 4–6× over the next 5–8 years, the urgency is clear. The question isn't whether inference hardware needs rethinking — it's how fast the industry and research community can deliver it.
